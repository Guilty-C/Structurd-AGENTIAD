What Changed
- Added an opt-in `first_turn_protocol_gate_mode` runtime surface with `off` and `retry_once_pz_cr`.
- Wired an auditable first-turn gate into the unified `baseline.py` tool loop for valid `pz_cr` runs only.
- When the first model turn still collapses to a direct final answer, the runtime now preserves that raw output, appends one deterministic protocol reminder, retries exactly once, and records the outcome as `recovered_to_tool_call`, `still_terminal_after_retry`, or `retry_parse_failure`.
- Added per-sample gate fields, `first_turn_gate.turn_0.json` sidecars, and aggregated `first_turn_gate_summary` blocks in summary / manifest / metrics.
- Added Prompt 2.7 unit tests and acceptance coverage.

Why It Changed
- Prompt 2.6 results show wording changes had zero behavioral effect.
- The `pz_cr` prompt surface is already valid, and train-side pz_cr supervision is strongly tool-first.
- This patch does not claim to fix the model. It is a minimal runtime experiment to answer whether one auditable protocol reminder plus exactly one retry can recover tool usage.

Effects
- The evaluator can now compare gate-off vs gate-on behavior without creating a second runtime stack.
- Gate behavior is fully auditable: original first response is preserved, retry is bounded to one, and no synthetic tool call is inserted.
- Run-level artifacts can now report whether the checkpoint recovers to tool usage or still collapses after the retry.
