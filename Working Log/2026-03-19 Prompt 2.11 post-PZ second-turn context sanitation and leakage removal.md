## What Changed

Prompt 2.11 keeps `baseline.py` as the single evaluation entrypoint and adds a narrow post-PZ second-turn sanitation step inside the existing gate-on `pz_cr` runtime path.

- added deterministic sanitation of the active message history immediately before the first assistant turn after successful `PZ` reinsertion
- removed only superseded assistant messages from the failed pre-gate terminal branch; user messages, system instructions, valid tool requests, and valid tool results remain intact
- extended `post_pz_transition.turn_*.json` sidecars with pre/post sanitation prompt surfaces, digests, removed-message evidence, and post-sanitization contract validity
- added prediction-level sanitation fields plus run-level sanitation summaries and grouped sanitation artifacts
- added Prompt 2.11 regression tests and acceptance audit coverage

## Why

Prompt 2.10 showed that the main near-term blocker was stale `pz_only`-era assistant leakage contaminating the second-turn prompt surface after successful first-turn `PZ` recovery. The next thin-layer runtime step was therefore to clean the active post-PZ conversation state, not to widen parser repair and not to force `CR`.

This patch does not force CR behavior. It removes only the obsolete assistant branch that was superseded by successful gate recovery so the next assistant turn is rendered from the runtime-canonical path.

## Effects

- original raw outputs for the failed first attempt and retry attempt remain preserved on disk
- the active second-turn rendered surface is now auditable both before and after sanitation
- runs now report how often sanitation was applied, how many obsolete assistant messages were removed, whether `pz_only` leakage survived after sanitation, and whether the post-sanitized CR contract became valid
- remote smoke/full runs can now answer whether CR remains zero after the post-PZ surface has been cleaned, rather than while stale pz-only contamination is still present
