# 2026-03-16 Prompt 2.8 first-turn gate retry tool-call repair for recoverable PZ-intent malformed outputs

## What Changed

- Added a bounded retry-only repair helper in `src/agentiad_recon/tooling.py` for malformed first-turn gate retry outputs that show one unambiguous PZ crop intent.
- Wired that helper into the existing `baseline.py` first-turn gate path only when the strict retry parse would otherwise fail under a valid `pz_cr` contract.
- Extended prediction records, run summary, run manifest, and metrics report with explicit repair attempt/success fields and aggregate repair counters.
- Added Prompt 2.8 tests and a dedicated acceptance script.

## Why It Changed

Prompt 2.7 already showed that the first-turn protocol gate can recover real `PZ` calls, but the remaining retry parse failures were not random noise. They were recoverable malformed PZ-intent outputs: missing wrappers, smart quotes, minor key punctuation corruption, and alias misuse like `tool_name="pz_cr"` with an explicit crop payload.

This patch targets that narrow gap only. It does not globally relax the parser, and it does not fabricate tool calls, CR usage, or bbox coordinates.

## Effects

- Recoverable retry-only malformed PZ outputs can now be normalized back into the existing strict `<tool_call>` contract.
- Unrecoverable pseudo-tool payloads still fail with concrete `failure_reason`.
- Original retry raw text and repaired canonical text are both preserved in auditable sidecars.
- Run-level repair aggregates make it easy to quantify how much of the old Prompt 2.7 retry-parse-failure bucket was truly recoverable malformed PZ-intent.

## Notes

- This patch is only for recoverable malformed PZ-intent retry outputs.
- It keeps strict non-retry parser behavior unchanged.
- It does not claim to fix the model or to recover `CR`; it only converts a bounded subset of gate-retry parse failures into auditable valid `PZ` calls when intent is explicit.
