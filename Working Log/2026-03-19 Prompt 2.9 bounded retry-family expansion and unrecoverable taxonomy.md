# 2026-03-19 Prompt 2.9 bounded retry-family expansion and unrecoverable taxonomy

## What Changed

- Extended the existing `pz_cr` first-turn gate retry repair helper in `src/agentiad_recon/tooling.py` so it can recover one unique malformed PZ crop-call candidate from bare JSON, wrapped JSON, or duplicated identical plain-plus-wrapper/plain-plus-`<think>` variants.
- Added bounded duplicate-candidate deduplication, literal-eval fallback for structured quote/list/tuple corruption, lossless `pz` / `PZ` / `pz_cr` alias handling, and lossless bbox canonicalization for dict, list/tuple, and four-scalar-key shapes.
- Added explicit unrecoverable retry families for pseudo-observation payloads, ambiguous multiple candidates, bbox-not-lossless cases, unsupported payload shapes, and no-unique-candidate failures.
- Propagated retry repair evidence into prediction records, gate sidecars, trace metadata, summary / manifest / metrics aggregates, and a small `metrics/first_turn_gate_repair_failure_families.json` artifact.
- Added Prompt 2.9 regression tests and a dedicated acceptance audit script.

## Why It Changed

Prompt 2.8 already proved that some gate-retry malformed outputs are recoverable, but the remaining bucket was still too coarse. The repo needed one more thin runtime/parser step that recovers clearly safe malformed PZ-intent outputs without relaxing the main parser, while also separating the unrecoverable remainder into auditable failure families instead of one vague generic failure.

This patch stays inside the existing evaluation/runtime layer. It does not touch trainer logic, SFT export semantics, canonical trajectories, rewards, or CR behavior shaping.

## Effects

- More first-turn gate retry outputs with one explicit malformed PZ crop intent can now recover into the existing strict `<tool_call>` contract.
- Strict parsing remains unchanged outside the existing gate-triggered retry parse-failure branch.
- Original retry raw text is preserved for every repair attempt, repaired canonical text is written only on success, and unrecoverable retries now carry explicit failure-family labels end to end.
- Run-level artifacts can now audit both repair success categories and the remaining unrecoverable families with zero missing failure reasons in the repair-failure summary.
