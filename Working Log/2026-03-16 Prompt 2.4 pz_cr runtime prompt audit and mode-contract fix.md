What Changed

- Added first-turn prompt-audit sidecars in the existing eval/runtime path so each tool-enabled sample records the actual rendered system text, user text, runtime tool mode, declared available tools, `pz_only` leakage flags, and CR-availability flags before generation.
- Added fail-fast mode-contract validation for `pz_cr` inside the unified `baseline.py` tool loop. If the rendered prompt surface leaks `pz_only` semantics or fails to expose CR, the sample now fails audibly before backend generation instead of pretending it was a valid `pz_cr` eval.
- Promoted `failure_reason` into the prediction-record contract and guaranteed that failed samples no longer keep `failure_reason = None`.
- Aggregated prompt-audit counters into metrics, run summary, and run manifest artifacts.

Why It Changed

- The authoritative remote full run already proved checkpoint loading and end-to-end artifact emission were real, but it also exposed the actual blocker:
  - full `pz_cr` eval finished end-to-end
  - checkpoint loading worked
  - artifacts were emitted
  - the model did not actually use tools
  - raw outputs in the supposed `pz_cr` run still contained `pz_only` wording
  - aggregate metrics showed `toolcall_rate = 0.0`, `PZ = 0.0`, and `CR = 0.0`
- That means the blocker is not training and not checkpoint loading. The blocker is eval/runtime mode-contract mismatch.

Effects

- The runtime now persists auditable prompt-audit sidecars from the real evaluation path.
- `pz_cr` no longer silently continues when the model-facing prompt surface is effectively `pz_only` or missing CR.
- Failed prediction records now always carry a concrete machine-checkable `failure_reason`.
- Run-level artifacts now expose prompt-audit mismatch counts and explicitly report zero missing failure reasons after the patch.
