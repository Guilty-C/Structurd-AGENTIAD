Timestamp: 2026-03-14
Prompt Number: Prompt 2.2

What changed
- Tightened `TransformersVisionLanguageBackend` device inference so it prefers the real language-input path (`get_input_embeddings`, `embed_tokens`, `language_model`, `base_model`, PEFT wrapper children) over misleading wrapper-level device shortcuts
- Expanded batch movement to cover generic mapping / BatchEncoding-like containers, nested dict/list/tuple payloads, and tensor-like `.to(...)` objects before `model.generate(...)`
- Added Prompt 2.2 tests that simulate a PEFT wrapper whose top-level device is misleading while the text-input embedding device is on CUDA
- Added Prompt 2.2 acceptance checks and a short README note

Why it changed
- Prompt 2.1 removed the deterministic generation warning, but the remote real smoke still reported `input_ids` on CPU while the model was on CUDA
- That meant Prompt 2.1's fix was incomplete: it improved generic device movement, but it still trusted wrapper-level device signals too early and did not explicitly pin the real language-input device path
- Prompt 2.2 fixes the runtime behavior itself instead of suppressing or filtering the warning

Effects
- The real checkpoint-eval path remains the same unified `baseline.py` entrypoint
- The backend now resolves the actual input-consuming device more robustly for PEFT/base wrappers
- The actual `encoded_inputs` passed into `model.generate(...)` are moved onto that target device before generation
- Deterministic generation hygiene from Prompt 2.1 remains intact
