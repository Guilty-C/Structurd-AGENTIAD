# 2026-03-19 Prompt 2.10 post-PZ second-turn CR transition audit under valid gate-on runtime

## What Changed

- Added a post-PZ second-turn transition audit in `src/agentiad_recon/baseline.py` that snapshots the actual prompt surface shown after the first successful `PZ` tool result is reinserted and before the next assistant generation happens.
- Added deterministic `post_pz_transition.turn_<n>.json` sidecars with prompt-surface digests, declared tool availability, reinserted-PZ evidence, CR-transition contract checks, mismatch reasons, and a classification of what the next assistant turn actually did.
- Extended prediction records, trace metadata, summary / manifest / metrics surfaces, and lightweight metrics artifacts with post-PZ transition audit fields and aggregate counts.
- Added Prompt 2.10 regression tests and a dedicated acceptance audit script.

## Why It Changed

Prompt 2.9 showed that first-turn PZ repair is no longer the main blocker. The runtime can now recover real `PZ` calls, but `CR` still stays at zero. The next thin-layer question is therefore diagnostic: after a real `PZ` call, does the second-turn prompt surface still expose a valid `CR` transition contract, or has the runtime/prompt surface degraded by then?

This patch answers that question with auditable runtime evidence only. It does not add a second-turn retry gate, does not coerce second-turn behavior, does not fabricate `CR`, and does not widen parser permissiveness.

## Effects

- The repo can now inspect exactly what the model saw on the next turn after `PZ`, including whether `CR` remained available and whether the `PZ` result was actually reinserted.
- The evaluator can now distinguish:
  - valid post-PZ CR contract but direct final answer without `CR`
  - invalid post-PZ CR contract where zero-`CR` behavior cannot yet be blamed purely on the model
- Run-level artifacts now expose post-PZ transition counts plus per-dataset and per-category slices without creating a second evaluator stack.
