## What Changed

Prompt 2.15 fixes the unified `transformers + peft` adapter-loading path inside the existing evaluator instead of creating a second runtime or sidecar evaluator.

The patch adds:

- explicit backend preflight via `prepare_runtime()` so an explicitly requested adapter is loaded before the evaluator starts burning through samples
- maintained PEFT attachment through `peft.PeftModel.from_pretrained(...)`
- attachment validation so `adapter_loaded=true` only when the returned runtime model exposes concrete PEFT state
- explicit adapter provenance fields across runtime summary/report/manifest surfaces:
  - `adapter_load_attempted`
  - `adapter_loaded`
  - `adapter_backend`
  - `adapter_load_error`
  - `adapter_load_error_type`
  - `adapter_load_error_repr`
  - `adapter_target_modules`
  - `allow_missing_adapter`
- an explicit CLI/runtime opt-out, `--allow-missing-adapter`, for runs that intentionally want base-only fallback while still preserving failure provenance
- tiny local regression coverage plus a Prompt 2.15 acceptance checker

## Why It Changed

The repo already had a checkpoint path threaded into runtime provenance, but real evaluation evidence showed the important distinction was broken:

- `adapter_checkpoint_path` was present
- `adapter_loaded` stayed `false`
- the checkpoint directory itself contained real PEFT adapter artifacts

That made the full 4-GPU evaluation output scientifically ambiguous, because the run looked adapter-aware on paper while still behaving like base-only inference.

The correct fix was inside the maintained runtime-loading seam, not in sharding, not in metrics, and not by introducing a second evaluator.

## Effect Of The Changes

- Runs with `--adapter-checkpoint-path` now fail fast by default if the adapter cannot actually be loaded.
- The same unified evaluator still owns baseline and tool-enabled inference; only adapter-load semantics changed.
- Runtime provenance can now distinguish:
  - adapter requested
  - adapter load attempted
  - adapter actually attached
  - adapter load failed but the run continued only because fallback was explicitly allowed
- Tiny smoke runs can now tell the user whether they are evaluating the intended SFT LoRA checkpoint or only the base model.
