# 2026-03-11 Prompt 1.8 truthful true-audit and resolved remote surfaces

- timestamp: 2026-03-11
- prompt number: 1.8

## What Changed

- Tightened Prompt 1.7 audit semantics so fallback-derived lengths cannot be reported as true-certified cleanliness.
- Extended Swift manifest and filtered manifests with explicit truth fields:
  - `true_multimodal_encode`
  - `length_audit_backend`
  - `threshold_clean_basis`
  - `strict_true_length_audit_requested`
  - `strict_true_length_audit_passed`
  - `true_threshold_clean_certified` (filtered manifests)
- Added resolved export surfaces artifact:
  - `prompt_1_8_resolved_remote_surfaces.json`
  - captures resolved dataset root, output root, model path, strict flag, backend, and filtered manifest paths.
- Kept strict mode behavior strict: `--strict-true-length-audit` fails when true multimodal encoder path is unavailable.
- Extended tests and acceptance checks to enforce truthful semantics and strict-mode behavior.

## Why It Changed

- Prompt 1.7 exposed a reporting gap: fallback audits were machine-usable but not explicitly separated from true multimodal encode certification.
- Prompt 1.8 requires this distinction to be encoded in artifacts so remote planning and smoke runs are evidence-based and auditable.

## Effects

- Non-strict export still emits usable subsets, but marks them as fallback-derived when true encoder was unavailable.
- Strict export now provides a hard guarantee boundary: no fallback masquerading as true certification.
- Remote execution no longer depends on unresolved template assumptions because resolved surfaces are persisted and referenced.
