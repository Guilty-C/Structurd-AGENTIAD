## What Changed

Prompt 2.14 adds deterministic sample sharding to the existing unified evaluator in `src/agentiad_recon/baseline.py` and adds one small shard-prediction merge helper in `src/agentiad_recon/merge_shards.py`.

The runtime now supports:

- `--num-shards`
- `--shard-index`

Sharding is applied only after the full stable sample list has already been enumerated, using:

`i % num_shards == shard_index`

The patch also threads shard metadata through runtime provenance, run summary, manifest, and metrics report surfaces:

- `num_shards`
- `shard_index`
- `full_sample_count`
- `selected_sample_count`
- `sharding_strategy = stable_index_mod`

## Why It Changed

The remote 4-GPU evaluation launcher was blocked because `baseline.py` did not expose shard arguments, so any attempt to run shard-specific jobs failed immediately with unrecognized CLI arguments.

The correct fix is a minimal deterministic sharding patch on the single evaluator entrypoint, not a second evaluator and not four copies of the same full run.

## Effect Of The Changes

- Default behavior remains unchanged when no shard arguments are passed.
- Sharded runs evaluate only their deterministic slice while preserving the same prompt, tool loop, parser, metrics definitions, and artifact semantics.
- Each shard now records enough metadata to show exactly which piece of the full sample set it evaluated.
- The merge helper can safely combine shard prediction JSONL files and rejects duplicate `sample_id` values instead of pretending to recompute aggregate metrics.
